# [C++] 다익스트라(Dijkstra) 시스템 크래시 방어: 오버플로우와 큐 메모리 누수 제어

대부분의 주니어 개발자들은 다익스트라(Dijkstra) 알고리즘을 단순한 '최단 경로 탐색 공식'으로 취급하며 맹목적으로 코드를 타이핑한다. 하지만 초고트래픽 서버의 라우팅이나 코어 백엔드 모듈에 이 알고리즘을 이식할 때, 방어 로직 없이 날것의 코드를 배포하면 즉시 **치명적인 메모리 누수(Memory Leak)와 시스템 크래시**에 직면하게 된다.

단순히 "코딩테스트 문제를 맞췄다"는 학부생 수준의 마인드셋을 버리고, C++ 환경에서 다익스트라를 구현할 때 발생하는 3가지 치명적인 시한폭탄과 이를 로우 레벨(Low-level) 관점에서 어떻게 방어하는지 그 트러블슈팅 과정을 기록한다.

---

##  1. 정수 오버플로우(Integer Overflow)의 공포

거리 배열(`dp`)을 초기화할 때 습관적으로 `INT_MAX`(`0x7fffffff`)를 사용하는 경우가 많다. 하지만 이는 C++ 환경에서 끔찍한 미정의 동작(Undefined Behavior)을 유발한다.

```cpp
// 위험한 코드: 시스템 크래시 유발
#define INF 0x7fffffff
dp[nxt_node] = dp[cur] + nxt_cost;
```
dp[cur]가 INF인 상태에서 새로운 간선의 가중치를 더하는 순간, 부호 비트가 침범당하며 값은 음수로 뒤집힌다. 라우팅 비용이 갑자기 마이너스가 되며 전체 네트워크 로직이 파괴되는 것이다.

* 방어 로직: 안전 마진 확보
```cpp
// 10억 언저리의 값으로, 덧셈 시 오버플로우 방지
const int INF = 0x3f3f3f3f;
```
0x3f3f3f3f를 사용하면 두 배를 더해도 INT_MAX를 넘지 않아 오버플로우를 원천 차단할 수 있다. 또한, memset을 통한 바이트 단위 초기화 시에도 값이 안전하게 보존되는 압도적인 이점이 있다.

## 2. 우선순위 큐 폭발 (Queue Blow-up) 제어
다익스트라의 성능은 우선순위 큐의 상태 관리에 달려있다. 간선을 순회하며 발견한 경로를 무지성으로 큐에 push하게 되면, 간선이 밀집된 그래프(Dense Graph)에서는 큐의 메모리가 기하급수적으로 터져버린다 (Memory Leak & TLE).
* 방어 로직: Stale Data 폐기 및 엄격한 간선 완화
시스템 메모리를 보호하기 위해서는 "기존 거리보다 확실히 짧은 경우에만 큐에 데이터를 적재"해야 하며, 큐에서 데이터를 꺼냈을 때도 이것이 과거의 쓰레기 데이터인지 검증하는 논리적 방화벽이 필수적이다.
```cpp
while(!pq.empty()) {
    auto [cost, cur] = pq.top(); pq.pop();

    // [1차 방화벽] 큐에서 꺼낸 정보가 이미 갱신된 과거의 쓰레기 데이터(Stale Data)라면 즉시 폐기
    if(dp[cur] < cost) continue; 

    for(auto nxt : vec[cur]) {
        int nxt_node = nxt.first;
        int nxt_cost = nxt.second;

        // [2차 방화벽] '간선 완화'가 발생하여 유의미한 최단 경로 갱신이 일어날 때만 큐에 적재
        if(dp[nxt_node] > dp[cur] + nxt_cost) {
            dp[nxt_node] = dp[cur] + nxt_cost;
            pq.push({dp[nxt_node], nxt_node});
        }
    }
}
```
단 한 줄의 if(dp[cur] < cost) continue; 로직이 낭비되는 CPU 사이클을 O(1)만에 차단하는 핵심 방어선이다.

## 3. 불필요한 메모리 대역폭 낭비 차단
경로 복원(Path Reconstruction)이 필요하지 않은 단순 비용 계산 로직에서, 큐에 tuple<int, int, int> 형태로 출발지 노드까지 무의미하게 저장하는 것은 메모리 대역폭의 낭비다.
* 방어 로직: 최소 페이로드 유지
pair<int, int>를 사용하여 오직 {누적 비용, 현재 노드}라는 핵심 페이로드만 들고 다님으로써 큐가 차지하는 메모리의 무게를 극단적으로 줄여야 한다.

## Conclusion
코어 엔지니어라면 코드가 정답을 내는 것을 넘어, 예외 상황에서 메모리가 터지지 않고 예상된 범위 내에서 안전하게 동작하는지를 로직으로 증명해야 한다.

불필요한 탐색을 차단하고 큐의 폭발을 방어하는 이 다익스트라 메모리 제어 감각은, 향후 C++ 비동기 서버의 트래픽 라우팅이나 락(Lock) 경합 최소화에도 동일하게 캐시 히트(Cache Hit)될 것이다.
